4. PPT – MAGYAR RÉSZ                               (~8 perc)
------------------------------------------------------------

[O mutatja: slide 1–2]                             (~1 perc)

[Slide 1 – Odett bevezeti:]
Ez a projekt neve és a két fejlesztő neve látható az első dián,
amit az előbb bemutattunk.

[Slide 2:]
A második dián látható az, amit az elején elmondtunk: miről szól

[P mutatja: slide 3–10]                            (~4 perc)

[Slide 3 – Patrik mondja:]
A tervezési fázisban először az adatbázis sémát és az alkalmazás
architektúráját kellett meghatározni. Öt fő entitást azonosítottunk,
amiket az ER-diagramon is láthatunk: users, categories, reports,
report_images és votes. Fontos volt a bejelentések státuszgépének
meghatározása: egy report háromféle állapotban lehet – pending,
approved vagy rejected. A felhasználói jogosultságrendszert
két role köré építettük fel: a sima user és az admin szerepkör.
Ezeket az entitásokat Eloquent modellek képviselik a kódban,
amik egymás között one-to-many és many-to-one relációkkal
vannak összekapcsolva.

[Slide 4:]
A backend technológiai stackje: Laravel 12 PHP 8.2 felett,
MySQL relációs adatbázis XAMPP alatt, Laravel Sanctum a stateless
Bearer token alapú autentikációhoz, és Eloquent ORM a database
absztrakciós rétegként. Az összes endpoint a localhost:8000/api
alap URL alatt érhető el.

[Slide 5:]
Az adatbázis táblák és az Eloquent kapcsolataik láthatók itt.
A reports tábla foreign key-ekkel kapcsolódik a users és
a categories táblához. A report_images és a votes táblák szintén
cascade delete relációban állnak a reports táblával – tehát
ha egy bejelentés törlődik, a hozzá tartozó képek és szavazatok
automatikusan szintén törlődnek, nem maradnak "árva" adatok.

[Slide 6:]
Az API endpointokat három hozzáférési szintre osztottuk.
Publikus, bejelentkezés nélkül elérhető végpontok: a jóváhagyott
bejelentések listája, a térkép adatok és a statisztikák –
tehát ezeket bárki láthatja. Védett végpontok, amelyekhez
bejelentkezés szükséges: bejelentés létrehozása, szerkesztése,
szavazás. Admin végpontok, amelyekhez admin jogosultság is kell:
moderáció, felhasználókezelés, kategória kezelés.

[Slide 7:]
A biztonság szempontjából egy háromrétegű middleware pipeline
van felépítve. Az auth:sanctum middleware validálja a Bearer
tokent és azonosítja a felhasználót. A check.banned middleware
403-as HTTP státuszkóddal blokkolja a bannolt felhasználók
kéréseit. Az admin middleware pedig role-alapon engedélyezi
vagy tiltja a hozzáférést az admin-only végpontokhoz.

[Slide 8:]
A képfeltöltés multipart/form-data content type-pal történik,
az images[] tömbmezőben. Server-side validáció gondoskodik arról,
hogy egyszerre maximum tíz fájl, darabonként maximum öt megabájt
méretben, és csak JPEG, PNG, GIF vagy WEBP MIME-típusban
érkezzen kérés.

[Slide 9:]
A szavazás idempotens logikával működik: ha egy felhasználó
ugyanolyan vote_type értékkel szavaz újra, a szavazat forceDelete-tel
törlődik – azért forceDelete-tel és nem soft delete-tel, mert
a unique constraint miatt egy soft deleted rekord is blokkolná
az újraszavazást. Ha ellentétes vote_type érkezik, a rekord
update-elődik. A credibility score az upvote-ok és downvote-ok
számának különbsége.

[Slide 10:]
Az admin a moderációs felületen tudja kezelni a bejelentéseket:
jóváhagyás, elutasítás és soft delete. A user management felületen
kereshet felhasználókra, és ban, illetve unban műveletet hajthat
végre – adminokat nem lehet bannolni, ezt server-side validáció
ellenőrzi. A tesztelés Postman collectionnel és PHPUnit alapú
automatizált feature tesztekkel történt, amelyek az AUTH, REPORT,
VOTE és ADMIN folyamatokat fedik le.


[O mutatja: slide 11–16]                           (~3 perc)

[Slide 11:]
A frontend technológiai stackje: Angular 19, teljesen standalone
komponens architektúrával – nincs NgModule, minden oldal önálló
lazy-loaded komponens. A UI Bootstrap 5.3.3 alapján készült,
CSS custom property-k felülírásával egy sci-fi stílusú sötét
témát alakítottunk ki. A térképhez Leaflet.js-t, reaktív
állapotkezeléshez Angular Signals-t, a munkamenet tárolásához
pedig sessionStorage-t használunk.

[Slide 12:]
Az Angular Router konfigurációjában minden route lazy-loaded:
a loadComponent callback csak akkor fut le, amikor a felhasználó
ténylegesen navigál az adott útvonalra, ami csökkenti az initial
bundle méretet. A ReportForm komponens két route-on is aktív:
a /reports/create és a /reports/:id/edit útvonalon – egy isEdit
flag alapján dönti el, melyik módban fusson. Az admin oldalakat
adminGuard védi, tehát csak admin role-lal rendelkező,
bejelentkezett felhasználó érheti el.

[Slide 13:]
Két CanActivate guard védi a privát route-okat: az authGuard
bejelentkezés hiányában a /login oldalra irányít át UrlTree-vel,
az adminGuard nem admin felhasználót a főoldalra irányít vissza.
Az authInterceptor egy funkcionális HttpInterceptorFn, amely
minden kimenő HTTP kérés headerjébe automatikusan beilleszti
az Authorization: Bearer token értéket. Az AuthService Angular
Signal-ban tárolja a currentUser objektumot, ami azonnal propagálja
a változást minden komponensbe, ahol hivatkoznak rá.

[Slide 14:]
Az app-map egy újrafelhasználható standalone komponens, amely
két Input paraméteren alapulva működik. Ha a clickable Input
false, read-only megjelenítő módban fut – csak jelzi a koordinátát
egy Leaflet markerrel. Ha clickable true, a map click eseményre
feliratkozva az EventEmitteren keresztül visszaadja a kiválasztott
[lat, lng] tömböt a szülő komponensnek. Az NgZone.runOutsideAngular
használata megakadályozza a felesleges change detection ciklusokat,
az invalidateSize() hívás pedig a lazy-loaded konténer méret
korrekcióját oldja meg.

[Slide 15:]
A ReportForm ugyanaz a komponens a létrehozáshoz és szerkesztéshez –
az ActivatedRoute snapshot-ból kiolvasott id paraméter alapján
dönti el az isEdit flag értékét. Template-driven Forms alapú
validációt alkalmazunk: kötelező mezők, maximum karakterszám,
és dátum jövőbeli érték tiltása. Képfeltöltésnél FileReader
API-val generálunk Data URL előnézetet. Szerkesztés módban a
meglévő images lista jelenik meg, ahol az egyes képek külön
DELETE kéréssel törölhetők az API-tól.

[Slide 16:]
A statisztika oldal publikus endpointon keresztül összesített
adatokat jelenít meg: jóváhagyott bejelentések száma, regisztrált
felhasználók, leadott szavazatok száma. A kategória szerinti
bontásnál arányos progress bar látható, ahol a legnagyobb
értéket 100%-nak vesszük és a többi ehhez képest skálázódik.
Az admin statistics endpoint részletesebb, minden státuszra
lebontott adatokat szolgáltat, beleértve a bannolt felhasználók
számát is.

[Bejelentkezés és validáció]
Most a backend logikáját mutatom meg a weboldalon
keresztül. Először nézzük az autentikációt – ha hibás credentialeket
küldünk a POST /api/login endpointnak, a backend Laravel Form
Request validációval hibát küld vissza – ez a 422-es HTTP státuszkód,
ami azt jelenti, hogy a kérés megérkezett, de az adatok érvénytelenek –,

azt a tokent automatikusan elküldi a szervernek – tehát
a szerver így tudja, hogy ki küldi a kérést, és van-e joga hozzá.

Ha egy bannolt felhasználó megpróbál belépni, a backend már
a belépési pontnál visszautasítja a kérést – tehát be sem
tud lépni, hiába tudja a jelszavát.

6. PPT – ANGOL RÉSZ                                (~3 perc)

[O mutatja: slide 17–18]                           (~1.5 perc)

[Slide 17 – Odett mondja:]
This is a brief English overview of our project. We built a
community-driven web application where users can submit reports
about UFO sightings and paranormal activities. Each report can
include GPS coordinates, a description, a category, witness count,
and optionally attached photos. Other authenticated users can vote
on the credibility of each report using an upvote and downvote
system, which produces a credibility score calculated as the
difference between upvotes and downvotes. Submitted reports go
through a content moderation workflow: they receive a pending
status by default and only appear publicly after an admin
approves them. The tech stack is Laravel 12 REST API on the
backend and Angular 19 SPA on the frontend.

[Slide 18:]
On the backend side, we used Laravel Sanctum for stateless
Bearer token authentication. We implemented role-based access
control with two roles: regular user and admin. The middleware
pipeline consists of three layers: token validation, ban check,
and role verification. Image upload is handled via multipart
form-data with server-side MIME type and file size validation,
and files are served from Laravel's public storage via a
storage symlink.

[P mutatja: slide 19–20]                           (~1.5 perc)

[Slide 19 – Patrik mondja:]
On the frontend side, we used Angular 19 with a fully standalone
component architecture and route-level lazy loading, which means
each page module is only fetched when the user navigates to it,
keeping the initial bundle size minimal. The interactive map
is built with Leaflet.js, a lightweight open-source mapping
library using OpenStreetMap tiles – no API key required. Route
access is protected by functional CanActivate guards, and a
functional HttpInterceptorFn automatically attaches the Bearer
token to every outgoing HTTP request.

[at the end]
That concludes our presentation. Thank you for your attention,
we are happy to answer any questions.
